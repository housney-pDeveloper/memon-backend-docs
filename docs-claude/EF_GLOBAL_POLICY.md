# EF_GLOBAL_POLICY.md

이 문서는 EF 전체 아키텍처의 철학과
각 서버의 역할을 정의하는 전사 선언서다.

이 문서는 모든 프로젝트의 CLAUDE.md보다 상위 철학 문서다.
단, 구현 세부 규칙은 각 프로젝트 CLAUDE.md가 우선한다.

---

==================================================
1. EF 아키텍처 철학
==================================================

EF는 역할이 명확히 분리된 다중 서버 구조를 가진다.

- Edge는 정책만 수행한다.
- Business는 도메인만 수행한다.
- Worker는 시스템 기능을 수행한다.
- Database는 구조만 책임진다.
- Common은 공유 규칙만 가진다.

각 계층은 자신의 책임만 가진다.
책임을 넘어서지 않는다.

---

==================================================
2. 서버 역할 정의
==================================================

--------------------------------------------------
2.1 Gateway (Edge Layer)
--------------------------------------------------

- 외부 요청의 단일 진입점
- JWT 인증 수행
- API 버전 정책 관리
- 라우팅
- 차단/Deprecated 정책 적용

하지 않는 것:
- 비즈니스 로직
- DB 접근
- 응답 가공

--------------------------------------------------
2.2 Application (Core Business)
--------------------------------------------------

- 실시간 사용자 요청 처리
- 모든 비즈니스 도메인 로직 수행
- 트랜잭션 관리
- 인가 처리
- 도메인 정책 집행

하지 않는 것:
- JWT 인증
- Gateway 역할
- 장시간 배치 처리 (부하 분리)

--------------------------------------------------
2.3 System (Worker Layer)
--------------------------------------------------

System은 Profile 기반으로 역할이 분리된다.

**api Profile:**
- 공통코드 API
- 알림 처리 (SSE)
- Health Check

**worker-mq Profile:**
- MQ 이벤트 소비 및 처리
- 로그 적재 (발송 로그, 시스템 로그)
- 외부 API 연동 (SMS, 알림톡 등)
- 이벤트 처리 결과 기록

**worker-batch Profile:**
- 배치 작업 실행
- 대용량 데이터 처리
- 정기 집계/동기화 작업
- 장시간 실행 허용

**공통 - 하지 않는 것:**
- 실시간 사용자 요청 처리 (Application 책임)
- JWT 생성/검증
- 사용자 인가 처리

--------------------------------------------------
2.4 Database (Schema Layer)
--------------------------------------------------

- 스키마 설계
- 제약 조건 정의
- 데이터 구조 유지

하지 않는 것:
- 비즈니스 정책
- 애플리케이션 의존 설계

--------------------------------------------------
2.5 ef-common (Shared Kernel)
--------------------------------------------------

- 공통 ErrorCode
- 공통 DTO
- 공통 유틸

하지 않는 것:
- 특정 서버 종속 로직

3. 통일 원칙
==================================================

3.1 응답 구조 통일

모든 서버는 동일한 응답 구조를 사용한다.

{
"code": number,
"message": string,
"data": object | null
}

---

3.2 에러 코드 체계 통일

형식:
{프로젝트번호}{비즈니스1}{비즈니스2}{시퀀스}

프로젝트 번호:

1 → Gateway
2 → System
3 → Application
9 → Common

---

3.3 보안 경계 (Trust Boundary)

- 외부 요청은 Gateway에서만 인증
- Application과 System은 Gateway 헤더만 신뢰
- 공유 비밀키 검증 필수

---

==================================================
4. 부하 분리 원칙
==================================================

4.1 실시간 트래픽 vs 배치 처리

- Application: 실시간 사용자 트래픽 전담
- System (worker-batch): 배치 작업 전담
- System (worker-mq): 비동기 이벤트 처리 전담

목적: Application 서버의 응답 지연 방지

---

==================================================
5. 스키마 분리 원칙
==================================================

5.1 스키마 구성

| 스키마 | 용도 | 주요 테이블 |
|--------|------|------------|
| gs_hr | 도메인 데이터 | TB_USER, TB_ORDER 등 |
| gs_log | 로그/이력 데이터 | TB_QUE_LOG, TB_SEND_LOG 등 |

---

5.2 스키마별 접근 권한

| 서버/Profile | gs_hr (도메인) | gs_log (로그) |
|-------------|----------------|---------------|
| Application | ALL | ALL |
| System (api) | SELECT | SELECT |
| System (worker-mq) | SELECT | SELECT, INSERT, UPDATE |
| System (worker-batch) | ALL | ALL |

---

5.3 스키마 분리 목적

- 최소 권한 원칙 적용
- 도메인 데이터 오염 방지
- 로그 테이블 독립 운영 (백업, 파티셔닝, 아카이빙)
- 보안 감사 시 명확한 접근 경계 증명

---

==================================================
6. 변경 원칙
==================================================

- Profile 간 책임은 명확히 분리한다.
- 한 서버의 기능 확장은 다른 서버의 책임을 침범하지 않는다.
- 경계가 모호하면 설계 재검토가 우선이다.

7. Claude 행동 원칙
==================================================

Claude Code는:

6. 스키마 경계를 준수한다.
